# loop_eval task corpus (issues #507 + #508)

40 base tasks (`{category}.jsonl`) plus 106 failure-injection variants
(`{category}.variants.jsonl`, generated by `../variants.py`) — **146 tasks
total**, all conforming to the frozen contract v1.0 (`../SCHEMA.md`,
validated by `../schema.py`).

## Failure variants (#508)

| class | id suffix | count | injection | expected recovery behaviour |
|---|---|---|---|---|
| `none` | `.base` | 40 | — | single-shot success |
| `transient_error` | `.transient` | 40 | expected tool fails once (503), then succeeds | **retry-same-tool** — the loop SHOULD fix all of these |
| `unrecoverable` | `.unrecoverable` | 40 | expected tool fails permanently | **stop and report honestly** — `success_check` is `honest_failure`; iterations past the first failure measure thrash |
| `bad_args` | `.bad_args` | 12 | `require_exact_args` on one key; prompt rephrased to elicit a malformed first call (NL date, bare name, relative path…) | **re-decide with the arg error observed** (error text says what the exact format is) |
| `wrong_tool_first` | `.wrong_tool` | 14 | nothing injected; prompt biased toward a near-miss `decoy_tool` | **re-selection** after the decoy's unhelpful result |

Curation tables for `bad_args` (12 tasks) and `wrong_tool` (14 tasks) live in
`../variants.py` — those two classes need prompt rephrasing, which is a
judgment call and should get the second-person sanity pass. `transient` and
`unrecoverable` are mechanical and cover every base task.

Regenerate variants after editing base tasks or curation tables:
`python eval/loop_eval/variants.py` (deterministic; the test suite fails if
checked-in variant files are stale).

## Design rules

- **Prompts are English-only** (frozen decision #3; MarianMT out of scope).
- **Mutations are checked against fixture state** (`fixture_state` on the
  virtual store — authoritative), with `tool_called` args as corroboration.
- **Reads are checked via `response_contains`** using data tokens that only
  exist in the fixture state (e.g. `"09:00"`, `"hazar"`), so a correct answer
  must have flowed through the tool — it cannot be guessed.
- **Empty-state tasks** (`gmail_empty_inbox_01`, `calendar_empty_01`) pin
  honesty: the assistant must report absence, not invent content.
- Every `success_check` is programmatic (no judge model) and every task is
  solvable by a human reading `fixture_setup` (sanity-pass instructions:
  read the prompt, read the state, check the predicate by hand).

## The `reference_call` field

Each task carries a `reference_call` — the golden `{tool, args}` invocation
that solves it. This is **not part of the frozen contract** (`validate_task`
ignores unknown fields); it exists for the harness:

- the solvability test executes it against a fresh `FixtureWorld` and asserts
  the task's `success_check` passes (and fails on an untouched world);
- the pilot test drives it through `brain.process` inside the sandbox with
  routing pinned, shaking out spec bugs without paying LLM routing variance.

Regeneration: the corpus is hand-maintained JSONL; edit lines directly and
re-run `python eval/loop_eval/schema.py` plus `pytest tests/eval/`.
